### 詳細実装仕様

#### 1. profiles RLSポリシー

```sql
-- 参照権限: status別制御
CREATE POLICY "profiles_select_policy" ON profiles
  FOR SELECT USING (
    CASE status
      WHEN 'active' THEN true
      WHEN 'suspended' THEN auth.uid() = id
      WHEN 'pending_deletion' THEN false
    END
  );

-- 更新権限: 本人のみ
CREATE POLICY "profiles_update_policy" ON profiles
  FOR UPDATE USING (auth.uid() = id);
```

#### 2. posts RLSポリシー

```sql
-- 参照権限: 複合条件
CREATE POLICY "posts_select_policy" ON posts
  FOR SELECT USING (
    status IN ('published', 'temporary') AND
    deleted_at IS NULL AND
    EXISTS (
      SELECT 1 FROM profiles
      WHERE id = posts.user_id AND status = 'active'
    ) AND
    CASE visibility
      WHEN 'public' THEN true
      WHEN 'friends' THEN (
        auth.uid() = user_id OR
        EXISTS (SELECT 1 FROM friends WHERE ...)
      )
      WHEN 'private' THEN auth.uid() = user_id
    END
  );
```

#### 3. media_files 継承ポリシー

- 親投稿の可視性を継承
- visibility優先度: private > friends > public > inherit

- [x] **authenticated**: SELECT/INSERT/UPDATE/DELETE ON 適切なテーブル
- [x] **サービスロール**: 管理APIのみで使用

### 10. 実装検証テストケース

#### 10.1 RLS有効化確認テスト

**テストID**: RLS-IMPL-001  
**テスト名**: 全テーブルRLS有効化確認テスト  
**目的**: 全対象テーブルでRLSが適切に有効化されていることの確認  
**優先度**: Critical  
**テストタイプ**: Integration

**テスト手順**:

```sql
-- 全テーブルのRLS有効状態確認
SELECT schemaname, tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
  AND tablename IN (
    'profiles', 'posts', 'media_files', 'follows', 'friends',
    'favorites', 'reposts', 'pinnings', 'post_tags', 'blocks'
  )
ORDER BY tablename;

-- 各テーブルのポリシー存在確認
SELECT schemaname, tablename, policyname, cmd, qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, policyname;
```

**期待結果**:

## 実装戦略

### 概要

BE-003タスクでは、RovRov SNSアプリのセキュリティの中核となるRow Level Security（RLS）ポリシーと権限設定を実装します。本実装戦略は、データベースレベルでのセキュリティを確保し、ユーザーデータの適切な分離と保護を実現することを目的としています。

実装は7つのフェーズに分かれており、各フェーズは前フェーズの成功を前提として段階的に構築されます。これにより、リスクを最小化しながら確実な実装を実現します。

### 技術的アプローチ

#### 1. RLSポリシー実装の原則

**階層的セキュリティモデル**

```
レイヤー1: ロール権限（GRANT）- 基本的なCRUD権限の制御
レイヤー2: RLSポリシー - 行レベルでの細かい制御
レイヤー3: DBトリガー - ビジネスロジックの強制（書き込み禁止等）
レイヤー4: アプリケーション - UIレベルでの制御
```

**ポリシー設計の優先順位**

1. **セキュリティファースト**: 漏洩リスクゼロを最優先
2. **パフォーマンス**: インデックス活用可能な条件設計
3. **保守性**: シンプルで理解しやすいポリシー構造
4. **拡張性**: 将来の機能追加に対応可能な設計

#### 2. 複雑なポリシーの実装アプローチ

**posts可視性の複合判定ロジック**

```sql
-- 判定優先順位（早期リターン最適化）
1. ブロック関係チェック（最優先、即座に拒否）
2. 投稿者status確認（pending_deletion/suspended）
3. 投稿status確認（deleted/draft）
4. visibility判定（public/friends/private）
5. 期限切れtemporary確認
```

**media_files継承ポリシー**

```sql
-- 継承ルール
1. visibility='inherit' → 親投稿のポリシーをそのまま適用
2. 独自visibility → MIN(親投稿visibility, メディアvisibility)
3. DBトリガーで親より広い設定を防止
```

#### フェーズ2: 基本RLSポリシー実装（1.5時間）

**目的**: シンプルなテーブルから順にRLSポリシーを実装

**実装順序と依存関係**:

```
profiles（独立）
  ↓
follows/blocks（profilesに依存）
  ↓

```

**実装内容**:

1. **profilesポリシー**

   ```sql
   -- 参照ポリシー
   CREATE POLICY "profiles_select" ON profiles
   FOR SELECT USING (
     CASE
       WHEN status = 'active' THEN true
       WHEN status = 'suspended' THEN auth.uid() = id
       WHEN status = 'pending_deletion' THEN auth.uid() = id
       ELSE false
     END
   );

   -- 更新ポリシー
   CREATE POLICY "profiles_update" ON profiles
   FOR UPDATE USING (auth.uid() = id)
   WITH CHECK (
     auth.uid() = id AND
     status = 'active'  -- suspendedユーザーは更新不可
   );
   ```

2. **followsポリシー**

   ```sql
   CREATE POLICY "follows_select" ON follows
   FOR SELECT USING (true);  -- 公開情報

   FOR INSERT WITH CHECK (
     follower_id = auth.uid() AND
     NOT is_blocked(follower_id, following_id)
   );
   ```

**検証ポイント**:

- 各statusでのprofileアクセス制御
- 本人のみ更新可能の確認

#### フェーズ3: 複雑な投稿RLSポリシー実装（2時間）

**目的**: posts/media_filesの複雑な可視性制御を実装

**実装内容**:

1. **postsポリシー（最重要）**

   ```sql
   CREATE POLICY "posts_select" ON posts
   FOR SELECT USING (
     -- 早期リターン：削除済み
     deleted_at IS NULL AND
     status NOT IN ('deleted', 'draft') AND

     -- 早期リターン：ブロック関係
     NOT EXISTS (
       SELECT 1 FROM blocks
       WHERE (blocker_id = auth.uid() AND blocked_id = posts.user_id)
          OR (blocker_id = posts.user_id AND blocked_id = auth.uid())
     ) AND

     -- 投稿者status確認
     EXISTS (
       SELECT 1 FROM profiles
       WHERE id = posts.user_id
         AND status = 'active'
     ) AND

     -- visibility判定
     CASE visibility
       WHEN 'public' THEN true
       WHEN 'friends' THEN
         auth.uid() = user_id OR
         EXISTS (
           SELECT 1 FROM friends
           WHERE status = 'accepted'
             AND ((user_id = auth.uid() AND friend_id = posts.user_id)
                  OR (user_id = posts.user_id AND friend_id = auth.uid()))
         )
       WHEN 'private' THEN auth.uid() = user_id
       ELSE false
     END AND

     -- temporary期限チェック
     (status != 'temporary' OR expires_at > NOW())
   );
   ```

2. **media_filesポリシー（継承実装）**
   ```sql
   CREATE POLICY "media_files_select" ON media_files
   FOR SELECT USING (
     deleted_at IS NULL AND
     status = 'ready' AND

     -- 親投稿の可視性を継承
     EXISTS (
       SELECT 1 FROM posts
       WHERE posts.id = media_files.post_id
         AND (
           -- ここでpostsのRLSポリシーが自動適用される
           CASE media_files.visibility
             WHEN 'inherit' THEN true  -- 親のRLSに完全依存
             WHEN 'public' THEN posts.visibility = 'public'
             WHEN 'friends' THEN posts.visibility IN ('public', 'friends')
             WHEN 'private' THEN true  -- 親のRLSで制御済み
           END
         )
     )
   );
   ```

**検証ポイント**:

- 各visibility設定での適切なアクセス制御
- ブロック関係での双方向制限
- temporary投稿の期限切れ処理
