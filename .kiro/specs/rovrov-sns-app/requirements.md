# Requirements Document

## はじめに

RovRovは、日常の小さな幸せを写真や動画として記録し、地図とタイムラインで振り返ることができる位置情報ベースのSNSアプリケーションです。ユーザーが日常を旅行のように楽しみ、新しい体験への一歩を踏み出すきっかけを提供することで、充実感のある生活をサポートします。

本アプリは「日常の充実感不足」と「行きたい場所が見つからない」という2つの主要な課題を解決し、飾らない日常の記録と、実際の体験に基づいた場所の発見を可能にします。

### 位置情報方針
- 投稿（Post）はPinが未設定でも作成可能
- 投稿は後からPinの追加・変更・削除が可能
- 地図に載せるか（&精度）は投稿ごとのユーザー設定で制御：
  - map_visibility: none（既定／地図に出さない）｜approx_100m（おおよそ100mで丸め）｜exact（正確）
- 地図に載せる座標はposts.display_point GEOGRAPHY(POINT,4326)に保持
- ベース座標の優先順位：Pin → EXIF/端末GPS → （なければ無し）
- map_visibilityに応じてdisplay_pointを生成／更新（noneならNULL）
- Pinは丸めない（施設は公知情報）。プライバシーは投稿側の地図表示設定で担保

## Requirements

### 要件1: ユーザー認証とアカウント管理（Supabaseマジックリンク認証）
**ユーザーストーリー:** ユーザーとして、パスワードなしで安全にアカウントを作成し、プロフィールを管理したい。これにより、自分の投稿や活動を記録し、他のユーザーとつながることができる。

#### 受け入れ基準

1. WHEN ユーザーが新規登録を開始する THEN システムは メールアドレスとユーザー名の入力フォームを表示しなければならない
2. WHEN ユーザーが新規登録でメールアドレスを送信する THEN システムは Supabaseを通じてマジックリンクをメールで送信しなければならない
3. IF ユーザー名が既に使用されている THEN システムは エラーメッセージを表示し、別のユーザー名の入力を促さなければならない
4. WHEN ユーザーがログインを開始する THEN システムは メールアドレスまたはユーザーIDの入力フォームを表示しなければならない
5. IF ユーザーIDが入力された THEN システムは 対応するメールアドレスを取得し、マジックリンクを送信しなければならない
6. WHEN ユーザーがマジックリンクをクリックする THEN システムは 認証を完了し、タイムライン画面へ遷移しなければならない
7. IF マジックリンクの有効期限が切れている THEN システムは エラーメッセージを表示し、再送信オプションを提供しなければならない
8. WHERE プロフィール編集画面で THE システムは ヘッダー画像、サムネイル、ユーザー名、自己紹介の編集機能を提供しなければならない
9. WHEN ユーザーがプロフィールを更新する THEN システムは 変更を保存し、確認メッセージを表示しなければならない
10. WHEN マジックリンクを開く THEN システムは iOS Universal Links / Android App Links でアプリに遷移しなければならない
11. WHERE マジックリンク THE 有効期限30分・単回使用で、失効後は再送オプションを提供しなければならない
12. IF 存在しないメールが入力された THEN システムは 常に同一の成功応答を返し、メール列挙を防止しなければならない

### 要件2: 投稿機能（Post）
**ユーザーストーリー:** ユーザーとして、日常の瞬間を簡単に写真や動画で記録し、場所情報と共に投稿したい。これにより、自分の体験を振り返り、他のユーザーと共有できる。

#### 受け入れ基準

1. WHEN ユーザーが投稿ボタンを押す THEN システムは カメラロールからの選択または撮影のオプションを提示しなければならない
2. IF ユーザーが撮影を選択した THEN システムは 複数枚の撮影を許可し、選択画面を表示しなければならない
3. WHEN 写真/動画が選択される THEN システムは GPSまたはEXIFデータから位置情報を自動取得し、pin（場所タグ）の候補を表示しなければならない
4. WHERE 投稿情報入力画面で THE システムは pin（場所タグ）、caption、tag、music、time、publicationの設定を可能にしなければならない
5. WHEN ユーザーが公開範囲を設定する THEN システムは Public（全員）、Friends（友達のみ）、Private（自分だけ）の選択肢を提供しなければならない
6. WHEN ユーザーがメディア単位で公開範囲を設定する THEN システムは 'inherit'・'public'・'friends'・'private' を選択可能にしなければならない
7. IF メディアの公開範囲が投稿の公開範囲より広い場合 THEN システムは保存を拒否し、「親より公開範囲を広げることはできません」とエラーメッセージを表示しなければならない
8. WHERE 表示において THE システムは 閲覧者に可視なメディアのみを表示し、不可視メディアの枚数をチップで示さなければならない
9. WHEN ユーザーがリポスト/共有する THEN システムは 'public' なメディアのみ許可しなければならない
10. IF ユーザーが下書き保存を選択した THEN システムは 投稿を下書きとして保存し、後で編集可能にしなければならない
11. WHEN 投稿が公開される AND 24時間が経過する THEN システムは ユーザーに本投稿として残すかの確認を求めなければならない
12. IF ユーザーが本投稿として残さない選択をした THEN システムは 投稿をフィードから削除し、journalにのみ保持しなければならない
13. WHEN 投稿を公開する THEN システムは 「地図表示が approx_100m を選ばれた場合」表示用座標を~100mグリッドへ丸めなければならない
14. WHEN 投稿が作成される THEN システムは Pin未設定でも可で `draft → temporary(24h) → archived|published` の状態管理を行わなければならない
15. WHEN ユーザーが投稿を作成する THEN システムは Pin未設定のまま下書き/公開ができなければならない
16. WHEN 投稿を公開する THEN システムは 「地図に表示」設定（none | approx_100m | exact）を提供し、既定は none としなければならない
17. IF 「地図に表示」が exact の場合 THEN システムは 正確な座標で地図に表示しなければならない
18. IF 「地図に表示」が approx_100m の場合 THEN システムは 約100mに丸めた座標で地図に表示しなければならない
19. IF 「地図に表示」が none の場合 THEN システムは 地図に表示しない（タイムライン等には表示）
20. WHEN 投稿作成画面で THE システムは EXIF/端末位置を用いて 近くのPin候補を提示しなければならない（許諾時）
21. WHEN 投稿後にユーザーが編集する THEN システムは Pinの追加・変更・削除を許可しなければならない
22. WHEN ユーザーが地図表示設定を変更する THEN システムは display_point を再計算し、地図表示を更新しなければならない

### 要件3: マップ機能（Rove）
**ユーザーストーリー:** ユーザーとして、現在地や興味のある場所周辺のおすすめスポットを地図上で発見したい。これにより、新しい場所を簡単に見つけて訪問できる。

#### 受け入れ基準

1. WHEN ユーザーがマップ画面を開く THEN システムは 現在地周辺のpinを地図上に表示しなければならない
2. WHERE マップ表示において THE システムは pinの名称のみを表示し、視認性を優先しなければならない
3. WHEN ユーザーがpinをタップする THEN システムは そのpinに紐づく投稿のサムネイル、名称、投稿内容、詳細ボタンを表示しなければならない
4. IF ユーザーが詳細ボタンを押す THEN システムは 所在地、営業時間、電話番号などの詳細情報を表示しなければならない
5. WHEN ユーザーが検索を実行する THEN システムは 店名や場所名でpinを検索し、結果を地図上に表示しなければならない
6. WHERE タグフィルター機能で THE システムは カフェ、景色、動物などのタグで投稿を絞り込む機能を提供しなければならない
7. IF ユーザーがpinをブックマーク（pinning）する THEN システムは そのpinを保存し、後でアクセス可能にしなければならない
8. WHILE マップを表示している間 THE システムは デフォルトでカフェタグのフィルターを適用しなければならない
9. WHERE マップ表示において THE システムは display_point IS NOT NULL の投稿のみを地図に表示しなければならない
10. WHEN 近傍投稿を検索する THEN システムは posts.display_point を用いて距離計算しなければならない

### 要件4: タイムライン機能（Timeline）
**ユーザーストーリー:** ユーザーとして、フォローしているユーザーや友達の日常的な体験をタイムライン形式で見たい。これにより、新しい場所の発見や、行きたい場所の記録ができる。

#### 受け入れ基準

1. WHEN ユーザーがタイムラインを開く THEN システムは ユーザー×日付でまとめたフィードを時系列で表示しなければならない
2. IF 同一ユーザーが同日に複数投稿している THEN システムは それらを1つのフィードにまとめ、横スクロール可能なカルーセル形式で表示しなければならない
3. WHERE タイムライン上で THE システムは featured（急上昇）、follow（フォロー中）、friend（友達）のタブを提供しなければならない
4. WHEN ユーザーが投稿にいいね（favorite）する THEN システムは いいね数をカウントし、表示を更新しなければならない
5. IF 投稿にpin情報がある THEN システムは pinningボタンを表示し、行きたい場所リストへの追加を可能にしなければならない
6. WHEN ユーザーがrepostする THEN システムは その投稿を自分のタイムラインに共有しなければならない
7. WHERE featuredタブで THE システムは 急上昇している人気の投稿を表示しなければならない
8. IF ユーザーが相互フォロー THEN システムは 友達申請の導線を提示しなければならない
9. WHERE 可視性衝突時（block等） THE システムは 最も厳しい制限を優先しなければならない
10. WHERE タイムライン表示で THE システムは 閲覧者に可視なメディアのみカルーセルに含め、非表示枚数をチップで示さなければならない

### 要件5: アカウント詳細画面（Account）
**ユーザーストーリー:** ユーザーとして、自分の投稿履歴を日記として振り返り、訪問した場所を地図上で確認したい。これにより、自分の活動記録を管理できる。

#### 受け入れ基準

1. WHEN ユーザーがアカウント画面を開く THEN システムは journal、rove、pinning、favoriteのタブを表示しなければならない
2. WHERE journalタブで THE システムは 自分の投稿を時系列で表示し、privateとpublicの表示切り替えを提供しなければならない
3. IF privateモードが選択された THEN システムは 全ての投稿（非公開含む）を表示しなければならない
4. IF publicモードが選択された THEN システムは 公開投稿のみを表示しなければならない
5. WHERE roveタブで THE システムは 地図に表示する意思がある投稿（display_point IS NOT NULL）のみをマップに描画しなければならない
6. WHERE pinningタブで THE システムは ブックマークした投稿を時系列順で一覧表示しなければならない
7. WHERE favoriteタブで THE システムは いいねした投稿を時系列順で一覧表示しなければならない
8. WHEN ユーザーがフォロー/フォロワー/友達リストを見る THEN システムは それらを自分だけに表示しなければならない
9. WHEN 投稿詳細を開く THEN システムは 地図表示設定とその出所（pin/exif/device/manual）を確認・変更できなければならない

### 要件6: ソーシャル機能
**ユーザーストーリー:** ユーザーとして、他のユーザーをフォローしたり、友達として登録したい。これにより、興味のあるユーザーの投稿を追跡できる。

#### 受け入れ基準

1. WHEN ユーザーが他のユーザーをフォローする THEN システムは そのユーザーの公開投稿をfollowタブに表示しなければならない
2. IF 相互フォロー状態になった THEN システムは 友達登録のオプションを提示しなければならない
3. WHEN 友達登録が完了する THEN システムは friendタブにそのユーザーの投稿を表示しなければならない
4. WHERE プライバシー設定で THE システムは フォロー数、フォロワー数、友達リストを本人のみに表示しなければならない
5. WHEN 友達関係が成立する THEN システムは 双方向の参照が成立し、相互に友達限定投稿を閲覧可能にしなければならない

### 要件7: pin（場所タグ）管理
**ユーザーストーリー:** ユーザーとして、新しいpin（場所タグ）を作成したい。これにより、まだ登録されていない場所も投稿に紐付けできる。

#### 受け入れ基準

1. IF 位置情報に該当するpin（場所タグ）が存在しない THEN システムは 新規pin（場所タグ）の作成オプションを提供しなければならない
2. WHEN ユーザーが新規pin（場所タグ）を作成する THEN システムは 名称と位置情報を保存し、他のユーザーも利用可能にしなければならない
3. WHERE pin（場所タグ）選択時に THE システムは 既存タグの候補と新規作成オプションを同時に表示しなければならない

### 要件8: データ永続性とパフォーマンス
**ユーザーストーリー:** ユーザーとして、投稿や設定が確実に保存され、アプリが快適に動作することを期待する。

#### 受け入れ基準

1. WHEN ユーザーがオフライン状態で投稿を作成する THEN システムは ローカルに保存し、オンライン復帰時に自動アップロードしなければならない
2. WHILE アプリを使用している間 THE システムは 画像の遅延読み込みとキャッシュを実装し、パフォーマンスを最適化しなければならない
3. IF データの同期エラーが発生した THEN システムは エラーメッセージを表示し、再試行オプションを提供しなければならない
4. WHERE 画像アップロード時に THE システムは 適切な圧縮を行い、ストレージとパフォーマンスを最適化しなければならない

## 非機能要件

### セキュリティ要件
1. WHEN ユーザーデータが送信される THEN システムは HTTPSを使用して暗号化しなければならない
2. WHERE 認証プロセスにおいて THE システムは Supabaseのマジックリンク認証を使用し、パスワードを保存しないようにしなければならない
3. IF 不正なアクセスが検出された THEN システムは アカウントをロックし、ユーザーに通知しなければならない
4. WHEN マジックリンクが生成される THEN システムは 有効期限（デフォルト30分）を設定し、一度使用されたリンクを無効化しなければならない

### データ保護要件
1. WHEN ユーザーが退会申請した場合 THEN システムは **即時は pending_deletion として完全不可視化（第三者不可視・書き込み不可・ログインのみ可）**し、**30日後に投稿・メディア・プロフィール（Storage/Stream含む）・auth.users を物理削除**しなければならない
2. WHERE 位置情報の生データ THE システムは 最短必要期間のみ保持（既定30日）しなければならない

### レート制限要件
1. WHERE 認証（OTP送信）において THE システムは 15分/5回（IP+メールの二軸）で制限しなければならない
2. WHERE API利用において THE システムは 1分/60リクエスト（ユーザー/匿名IPの双方に適用）で制限しなければならない

### 監視要件（MVP）
1. WHEN システムが稼働している THEN システムは 認証成功率、OTP再送率を監視しなければならない
2. WHEN システムが稼働している THEN システムは TL p95応答時間、画像変換p95、アップロード成功率を監視しなければならない

### 使いやすさ要件
1. WHEN 新規ユーザーが初めてアプリを起動する THEN システムは オンボーディングチュートリアルを表示しなければならない
2. WHERE 全ての画面において THE システムは 3タップ以内で主要機能にアクセス可能にしなければならない
3. WHILE 画像をアップロードしている間 THE システムは 進捗インジケーターを表示しなければならない

### 要件9: モデレーション（アカウント状態）
**ユーザーストーリー:** 管理者として、不適切な行為を行うユーザーのアカウントを適切に管理し、健全なコミュニティを維持したい。また、ユーザーとして、アカウント削除時はデータが適切に処理されることを期待する。

#### 受け入れ基準

1. WHEN ユーザー状態が **suspended** になった THEN システムは そのユーザーの**新規投稿・編集・削除**を拒否しなければならない
2. WHEN ユーザー状態が **suspended** になった THEN システムは そのユーザーの**既存投稿・メディアを他者には非公開**にし、本人は閲覧のみ可能にしなければならない
3. WHEN ユーザー状態が **deleted** になった THEN システムは **ログインを拒否**しなければならない
4. WHEN ユーザー状態が **deleted** になった THEN システムは 当該ユーザーの**投稿・メディア・プロフィールを物理削除**し、**auth.users も削除**しなければならない
5. WHERE タイムライン/マップ/検索表示において THE システムは status != 'active' のユーザーのコンテンツを**返さない**こと
6. WHERE プロフィール画面において THE システムは suspended 状態のユーザーに**「アカウントは一時停止中です」**バナーを表示しなければならない
7. WHERE 投稿・編集操作において THE システムは suspended 状態のユーザーの操作を**トーストメッセージで理由表示して拒否**しなければならない
8. WHERE 投稿・編集操作において THE システムは pending_deletion 状態のユーザーの操作を**トーストメッセージで理由表示して拒否**しなければならない
9. WHEN ユーザー状態が **pending_deletion** になった THEN システムは **ログインを許可**し、**プロフィールは第三者に不可視**（「アカウントが存在しません」と表示）、**新規投稿/編集/削除を拒否**し、**既存投稿/メディアは他者に不可視**にしなければならない
10. WHEN ユーザー状態が **pending_deletion** になってから **30日が経過**した THEN システムは **投稿・メディア・プロフィール（Storage/Stream含む）を物理削除**し、**auth.users も削除**しなければならない
11. WHEN **pending_deletion** のユーザーが **30日以内に復帰**を選択した THEN システムは profiles.status='active' に戻し、**過去の投稿表示を自動で復帰**させなければならない
12. WHERE タイムライン/マップ/検索表示において THE システムは profiles.status='active' のユーザーのコンテンツ**のみ**を返さなければならない（= suspended/pending_deletion/deleted は非表示）

## MVP除外項目

以下の機能はMVPでは実装しない：
- 通知機能（Notification）
- リアルタイムメッセージング
- 動画編集機能
- 外部SNSとの連携
