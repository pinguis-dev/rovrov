## 実装戦略

### 概要
BE-002は、RovRov SNSアプリケーションの基盤となるデータベーススキーマを構築する重要なフェーズです。既存のSupabase環境（BE-001で構築済み）上に、13の主要テーブル、5つのENUM型、8つの重要インデックスを作成し、位置情報ベースのソーシャル機能を実現するための堅牢なデータ構造を確立します。

### 実装アプローチ

#### 1. 段階的実装戦略
データベーススキーマの実装を以下の4段階に分けて実行します：

**Phase 1: 基盤構築（1-1.5時間）**
- ENUM型定義の作成
- 基本テーブル（profiles、posts）の作成
- 既存データの移行準備

**Phase 2: リレーション構築（1.5-2時間）**
- 関連テーブル（media_files、pins、follows、friends等）の作成
- 外部キー制約の設定
- CASCADE動作の検証

**Phase 3: 最適化実装（1-1.5時間）**
- 空間インデックス（GIST）の作成
- 複合インデックスの作成
- 部分インデックスの作成

**Phase 4: 検証・調整（0.5-1時間）**
- 全テーブル・制約の動作確認
- パフォーマンステスト
- ロールバック手順の確認

#### Step 3: 主要テーブルの作成

```sql
-- 3.1 profilesテーブル（auth.usersとの連携）
CREATE TABLE IF NOT EXISTS profiles (
  id UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  username VARCHAR(50) UNIQUE NOT NULL,
  email VARCHAR(255) UNIQUE,
  display_name VARCHAR(100),
  bio TEXT,
  avatar_url TEXT,
  header_url TEXT,
  status profile_status NOT NULL DEFAULT 'active',
  suspended_until TIMESTAMPTZ,
  moderation_note TEXT,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at TIMESTAMPTZ
);

-- 3.2 pinsテーブル（位置情報マスタ）
CREATE TABLE IF NOT EXISTS pins (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  name VARCHAR(255) NOT NULL,
  location GEOGRAPHY(POINT, 4326) NOT NULL,
  source VARCHAR(50) NOT NULL DEFAULT 'user',
  source_place_id VARCHAR(255),
  created_by UUID REFERENCES profiles(id) ON DELETE SET NULL,
  address TEXT,
  attribution_text TEXT,
  posts_count INTEGER NOT NULL DEFAULT 0,
  is_visible BOOLEAN NOT NULL DEFAULT true,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- 3.3 postsテーブル（投稿データ）
CREATE TABLE IF NOT EXISTS posts (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  pin_id UUID REFERENCES pins(id) ON DELETE SET NULL,
  caption TEXT,
  status post_status NOT NULL DEFAULT 'draft',
  visibility visibility_type NOT NULL DEFAULT 'public',
  map_visibility map_visibility_type NOT NULL DEFAULT 'none',
  display_point GEOGRAPHY(POINT, 4326),
  location_source location_source NOT NULL DEFAULT 'none',
  expires_at TIMESTAMPTZ,
  published_at TIMESTAMPTZ,
  favorite_count INTEGER NOT NULL DEFAULT 0,
  repost_count INTEGER NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  deleted_at TIMESTAMPTZ
);

-- 3.4 media_filesテーブル（メディアファイル）
CREATE TABLE IF NOT EXISTS media_files (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  post_id UUID NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  bucket VARCHAR(255) NOT NULL,
  path TEXT NOT NULL,
  thumb_path TEXT,
  type media_type NOT NULL,
  visibility visibility_type NOT NULL DEFAULT 'public',
  mime_type VARCHAR(100),
  file_size BIGINT,
  width INTEGER,
  height INTEGER,
  duration_seconds INTEGER,
  stream_uid VARCHAR(255),
  status media_status NOT NULL DEFAULT 'ready',
  display_order INTEGER NOT NULL DEFAULT 0,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
```

#### Step 4: ソーシャル機能テーブルの作成

```sql
-- 4.1 followsテーブル（フォロー関係）
CREATE TABLE IF NOT EXISTS follows (
  follower_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  following_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (follower_id, following_id),
  CHECK (follower_id != following_id)

-- 4.2 friendsテーブル（友達関係）
CREATE TABLE IF NOT EXISTS friends (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),

  friend_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  status friend_status NOT NULL DEFAULT 'pending',
  requested_by UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  accepted_at TIMESTAMPTZ,
  UNIQUE(user_id, friend_id),
  CHECK (user_id != friend_id)
);

-- 4.3 favoritesテーブル（いいね）
CREATE TABLE IF NOT EXISTS favorites (
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  post_id UUID NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, post_id)
);

-- 4.4 repostsテーブル（リポスト）
CREATE TABLE IF NOT EXISTS reposts (
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  post_id UUID NOT NULL REFERENCES posts(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (user_id, post_id)
);

-- 4.5 pinningsテーブル（ブックマーク）
CREATE TABLE IF NOT EXISTS pinnings (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  pin_id UUID REFERENCES pins(id) ON DELETE CASCADE,
  post_id UUID REFERENCES posts(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  CHECK (pin_id IS NOT NULL OR post_id IS NOT NULL)
);

-- 4.6 blocksテーブル（ブロック関係）
CREATE TABLE IF NOT EXISTS blocks (
  blocker_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  blocked_id UUID NOT NULL REFERENCES profiles(id) ON DELETE CASCADE,
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  PRIMARY KEY (blocker_id, blocked_id),
  CHECK (blocker_id != blocked_id)
);
```

#### Step 8: トリガー関数の作成

```sql
-- 8.1 updated_atの自動更新トリガー
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- 8.2 各テーブルへのトリガー適用
CREATE TRIGGER update_profiles_updated_at BEFORE UPDATE ON profiles
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_posts_updated_at BEFORE UPDATE ON posts
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

CREATE TRIGGER update_pins_updated_at BEFORE UPDATE ON pins
  FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();

-- 8.3 カウンタキャッシュ更新トリガー
CREATE OR REPLACE FUNCTION update_favorite_count()
RETURNS TRIGGER AS $$
BEGIN
  IF TG_OP = 'INSERT' THEN
    UPDATE posts SET favorite_count = favorite_count + 1 WHERE id = NEW.post_id;
  ELSIF TG_OP = 'DELETE' THEN
    UPDATE posts SET favorite_count = favorite_count - 1 WHERE id = OLD.post_id;
  END IF;
  RETURN NULL;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER update_post_favorite_count
AFTER INSERT OR DELETE ON favorites
FOR EACH ROW EXECUTE FUNCTION update_favorite_count();
```

### 技術的考慮事項

#### 1. PostGIS活用戦略
- **GEOGRAPHY型の採用**: 球面座標系での正確な距離計算を実現
- **GIST インデックス**: 空間検索の高速化（ST_DWithin、ST_Distance関数との連携）
- **座標精度管理**: map_visibilityに応じた座標丸め処理の実装

#### 2. パフォーマンス最適化
- **複合インデックスの列順序**: 検索頻度の高い列を先頭に配置
- **部分インデックス**: WHERE句付きインデックスによるストレージ削減
- **カウンタキャッシュ**: favorite_count、repost_countによるCOUNTクエリの回避

#### 3. データ整合性保証
- **外部キー制約**: CASCADE/SET NULLによる参照整合性の自動維持
- **CHECK制約**: 自己参照の防止（follows、friends、blocks）
- **UNIQUE制約**: username、email、複合キーによる重複防止

#### 4. 拡張性考慮
- **UUID採用**: 分散環境でのID衝突回避
- **JSONB型**: pin_attributesでの柔軟な属性管理
- **ENUM型**: 将来の値追加を考慮した設計

   ```sql
   LEFT JOIN media_files m ON p.id = m.post_id
   WHERE p.id = (SELECT id FROM posts ORDER BY created_at DESC LIMIT 1)
   GROUP BY p.id, p.status, p.published_at, p.display_point;
   
   COMMIT;
   ```

**期待結果**: 
- 投稿が正常に作成され、公開状態になること
- 関連メディアファイルが正しく紐づけられること
- display_pointが設定されること

**優先度**: Critical
